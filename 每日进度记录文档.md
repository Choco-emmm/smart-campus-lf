## 第五天

### 1. 项目进度

- 明确并落地了**“双轨制”状态管理架构**，严格区分了用户的“逻辑删除（`@TableLogic`）”与管理员的“违规下架（`status=3`）”。确保违规帖子从公共广场消失，但仍保留在用户的“我的发布”列表中以作案底警示。
- 实现了**举报审批与跨表联动机制**，管理员核实举报属实后，系统通过 `@Transactional` 事务自动将原帖状态修改为违规下架，并强制扒除其可能存在的“置顶”特权。
- 实现了**置顶申请的完整闭环**，支持动态条件分页查询置顶申请单；审批通过后自动修改物品主表状态并计算置顶过期时间（默认24小时），完美衔接前期的定时自动降级任务。
- 实现了**后台用户治理模块**：
  - 运用 MyBatis-Plus 的 `LambdaQueryWrapper` 优雅解决了状态筛选（`AND`）与多字段模糊搜索（嵌套 `OR`）的 SQL 拼接冲突问题。
  - 新增用户档案详情接口，通过 Mapper 动态执行 `COUNT` 统计用户的历史违规发帖次数，避免了数据冗余与一致性问题。
- **修复了安全漏洞**，在查询层和操作层双管齐下，严格拦截了试图查看或封禁同级管理员账号的越权请求，保障了系统底线安全。
- 实现了 **JWT 动态踢人与封禁机制**，创新性地复用 Redis 已有 Key，将封禁用户的 Token 值替换为 `banned` 并沿用原 TTL，配合 `TokenInterceptor` 拦截器改造。
- 落地了**前后台场景分离**设计，新增 “他人公开主页”接口，返回极致脱敏的 `UserProfileVO`，并巧妙保留 `status` 字段供前端渲染“已被封禁”提示，优化了用户间的社交沟通体验。
